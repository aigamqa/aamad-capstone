## 4. FSM Implementation Notes

The FSM is implemented in `app/dashboard/page.tsx` using `useReducer` with `FsmState` and `FsmAction` types.

| FSM State | Entry Mechanism | UI Rendered |
|-----------|----------------|-------------|
| `idle` | Initial state (no active run) | Centered card with "No active run" + Scan Now button |
| `running_scan` | SCAN_TRIGGERED dispatch | Blue Alert banner with spinner + two SkeletonCard components |
| `running_qualify` | RUN_STARTED with status="qualifying" | Same as running_scan (step label updates from runStatus) |
| `running_write` | RUN_STARTED with status="writing" | Same pattern, step 3/3 label |
| `cp1_pending` | CP1_FIRED dispatch | Cp1Gate component at top, no status banner |
| `cp2_ready` | CP2_READY dispatch; drafts loaded | Progress bar + DraftCard[] list |
| `done` | ALL_REVIEWED (reviewed_ids.length === drafts.length) | Centered summary Card with counts + Mark pipeline complete |
| `error` | RUN_ERROR dispatch | Red error card + Retry scan button |

SWR polling: `refreshInterval: 3000` when `shouldPoll` (running/idle states), `0` in cp2_ready/done/error. Polling key is null when shouldPoll is false, which stops SWR from fetching.

Default render state: `useEffect` on mount calls `getRunStatus()` which returns `status: "awaiting_cp2"` → FSM transitions to `cp2_ready` → `getDrafts()` loads mock 2-draft array. Dashboard opens in cp2_ready showing both draft cards.

## 5. assistant-ui Decision

**Decision: Textarea fallback used. assistant-ui NOT integrated.**

**What was tried**: `@assistant-ui/react@0.14.5` installed successfully via npm. Package exports `ThreadPrimitive`, `ExternalThread`, and related runtime primitives.

**Why it was not used**: `@assistant-ui/react` requires an `AssistantRuntime` context connected to a live LLM backend (Anthropic API key or equivalent). The MVP frontend is a stub-only scaffold with no backend wired. The `Thread` component cannot be embedded inline within a Card without a runtime provider. Attempting to use it would require wiring an API key and backend endpoint — which is the Integration epic's scope, not the Frontend epic.

**Traceability**: Spec Open Question #6 is unresolved. The assistant-ui inline embed mode has not been confirmed compatible with the card-based layout described in spec Section 8.

**Fallback implemented**: `components/draft-card/DraftEditor.tsx` uses a controlled `<textarea>` with:
- Placeholder: "AI assist pending — edit your proposal here"
- Comment in source: `// assistant-ui inline embed mode not confirmed (spec Open Question #6); using plain Textarea fallback`
- Save button calling `submitFeedback(draftId, "edit", content)`
- Scroll gate reset after save (user must re-read edited draft)

**Integration epic TODO**: Once FastAPI backend is wired, revisit `DraftEditor.tsx`. If `@assistant-ui/react` supports inline embedding with an `ExternalStoreAdapter`, use it. Otherwise maintain Textarea. Document the final decision in `integration.md`.

## 6. CP1 Gate Decision

**Decision: Collapsible Alert at top of page. NOT a blocking Dialog.**

**Spec Open Question #3 resolution**: The spec listed two options:
1. Full-screen blocking Dialog (stronger HITL enforcement)
2. Collapsible Alert section at top of page (less intrusive)

**Chosen**: Collapsible Alert. Rationale for MVP:
- shadcn Dialog component was not available in the installed shadcn version (base-nova theme used by shadcn@4.7.0 required `@base-ui/react` primitives incompatible with Next.js 14 + Tailwind CSS 3)
- Collapsible Alert still provides prominent visual treatment (amber color, AlertTriangle icon, pending count label)
- "Cannot navigate away" is enforced via `beforeunload` event handler — soft enforcement as spec Section 3.4 specifies
- All borderline job cards with Include/Skip buttons are rendered inside the Alert
- Submit button is disabled until all jobs are actioned
- For V1: if stronger enforcement is needed, replace with `@radix-ui/react-dialog` Dialog directly

**Traceability**: `components/cp1-gate/Cp1Gate.tsx` contains the comment: "MVP Decision: Implemented as a collapsible Alert at top of page, NOT a full Dialog."

## Assumptions

1. shadcn@4.7.0 (base-nova theme) generates components using `@base-ui/react` primitives and Tailwind CSS 4 syntax, which is incompatible with Next.js 14 + Tailwind CSS 3. All shadcn components were rewritten using standard Radix UI primitives and Tailwind CSS 3 utility classes.
2. `@assistant-ui/react@0.14.5` requires a live LLM backend runtime and cannot be embedded inline without one. Textarea fallback is used pending Integration epic.
3. Geist font via `next/font/google` is not available in Next.js 14.2 — using local GeistVF.woff fonts bundled by `create-next-app`.
4. `lucide-react@1.16.0` was already bundled transitively; explicitly added for icon imports in components.
